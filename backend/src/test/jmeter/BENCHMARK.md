# 笔记详情页多级缓存压测报告

## 1. 测试目标

验证 **Caffeine(L1) + Redis(L2)** 多级缓存对 `GET /api/notes/{noteId}` 接口的性能提升效果。

## 2. 测试环境

| 项目 | 配置 |
|------|------|
| 部署方式 | 单体部署（Spring Boot 2.7 + MySQL 8.0 + Redis 7.0） |
| 机器配置 | 本地开发机 / 4核8G 云服务器 |
| JVM 参数 | `-Xms1g -Xmx1g` |
| 网络 | 本地回环（127.0.0.1）或内网 |
| 数据规模 | note 表 10 万条记录 |
| 访问模式 | 80/20 法则：80% 请求集中在 Top 20% 热点 noteId |

> **注意**：以下“参考数据”来源于同类单体项目（Spring Boot + MyBatis + MySQL）在 4C8G 环境下的典型表现，**请在你的环境中实测后替换为真实数字**。

## 3. 前置准备

1. 启动 MySQL、Redis 和本地后端服务（端口 8080）。
2. 确保 `note` 表中有足够数据（建议至少 1,000 条以上，ID 连续分布）。
3. 使用 JMeter 打开 `NoteDetailTestPlan.jmx`。
4. 如需模拟 80/20 热点分布，可将路径中的 `${__Random(1,1000)}` 替换为 CSV 数据文件配置（前 200 个 ID 占 80% 请求）。

## 4. 测试步骤

### 4.1 优化前压测（回滚缓存代码）

临时将 `NoteServiceImpl.getNoteDetail()` 中的缓存逻辑注释掉，改为直接查库：

```java
Note note = noteMapper.findById(noteId);
```

运行 JMeter：
- 线程数：**100**
- Ramp-up：**5s**
- 持续时间：**60s**
- 记录聚合报告中的 **Samples、Average、Throughput、Error %**。

### 4.2 优化后压测（恢复缓存代码）

恢复 `NoteServiceImpl.getNoteDetail()` 的 L1 + L2 缓存逻辑，**预热缓存**（可先手动请求 10~20 次热点接口，或运行 10 秒预热线程组）。

再次运行相同参数的 JMeter 测试，记录数据。

## 6. 同类项目参考数据

> 以下数据仅供参考，用于判断你的测试结果是否处于合理区间。

在 **4核8G / 本地回环 / 100 并发 / 60 秒** 的条件下，典型表现如下：

| 指标 | 优化前（参考） | 优化后（参考） | 参考提升 |
|------|---------------|---------------|----------|
| 平均 RT | 45 ~ 80 ms | 5 ~ 12 ms | 约 5~8 倍 |
| 95% RT | 120 ~ 250 ms | 15 ~ 35 ms | 约 5~10 倍 |
| 吞吐量 | 150 ~ 350 QPS | 1200 ~ 2500 QPS | 约 5~10 倍 |
| MySQL CPU | 60% ~ 90% | 3% ~ 10% | 显著下降 |
| Caffeine 命中率 | — | 55% ~ 75% | — |
| Redis 命中率 | — | 20% ~ 35% | — |

**说明**：
- 若你的数据量很小（<1万条）且 MySQL 数据完全在 innodb_buffer_pool 中，优化前的 QPS 可能本身就较高（>500），缓存带来的倍数提升会相应减小。
- 若你的网络环境是外网或 Redis 在远程机器，L2 Redis 的 RT 会比本地 MySQL 快得有限，但 L1 Caffeine 的本地命中部分依然能贡献巨大提升。

## 7. 简历写法建议（基于实测数据）

> 针对笔记详情页高并发读场景，设计并落地 **Caffeine（L1）+ Redis（L2）多级缓存架构**。经 JMeter 压测（100 并发 / 60s），接口平均 RT 从 **XX ms** 降至 **XX ms**，吞吐量从 **XX QPS** 提升至 **XX QPS**，MySQL CPU 占用下降 **XX%**；通过 **空值缓存 + 互斥锁 + 随机 TTL** 解决缓存穿透、击穿、雪崩问题，采用 **Cache-Aside + 延迟双删** 保证缓存与数据库最终一致。
